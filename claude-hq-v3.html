// capacidades reutilizables

Skills

Una skill es una carpeta con un SKILL.md que le enseña a Claude un procedimiento o estándar del equipo. Claude la lee sola cuando la tarea lo amerita. Es la forma de que "no haya que repetirle las cosas".

estructura

Anatomía de una skill

mi-skill/
├── SKILL.md      ← obligatorio
├── ejemplos/     ← opcional
└── scripts/      ← opcional

# SKILL.md
---
name: mi-skill
description: Cuándo debe activarse
  (¡esto es lo más importante!)
---
Instrucciones paso a paso...

La description decide si Claude la usa: escribila pensando "¿qué frases diría alguien que necesita esto?"

plugin vs skill

Cómo se distribuyen

  • Una skill es el contenido (un SKILL.md). Un plugin empaqueta una o más skills (y a veces hooks o MCP) para instalarlas de un comando.
  • Un marketplace es un repo de GitHub que distribuye muchos plugins. El de Anthropic viene por defecto.
  • Scope al instalar: User (~/.claude/skills/, para todos tus proyectos) o Project (.claude/skills/, solo ese repo).
cómo instalar una skill

Método 1 — Marketplace de plugins (lo más rápido)

Abrí el navegador de plugins, andá a la pestaña Discover, elegí uno y Enter. Te pregunta el scope (User o Project) y queda activa.

Método 2 — Que la cree Claude (para skills propias del equipo)

La forma recomendada para las nuestras: le pasás el prompt y Claude genera la carpeta y el SKILL.md en .claude/skills/. Después se comparte con el equipo por Git.

Método 3 — Manual

Creás la carpeta a mano y pegás el SKILL.md adentro. Ojo: la carpeta va directo dentro de skills, sin un nivel extra.

bootstrap seguro

Prompt para Claude Code

Levanta el stack completo con la capa de seguridad aplicada desde el primer commit. Para arranque guiado por producto, usá el "Kickoff seguro 0→1" de /prompts.

capa de ciberseguridad — no negociable
  • RLS en cada tabla de Supabase antes de insertar el primer dato.
  • Validación con Zod en todo input: formularios, API routes, server actions.
  • Secrets solo en .env.local + Vercel; excluidos en .gitignore desde el commit 1.
  • Auth verificada en el servidor, nunca solo en el cliente.
  • Service role key jamás en el cliente; solo server-side.
  • Headers de seguridad en next.config: CSP, HSTS, X-Frame-Options, nosniff.
  • Rate limiting en endpoints sensibles (login, formularios públicos).
  • npm audit / Dependabot en CI: el build falla con vulnerabilidades críticas.
  • Sanitización de todo lo que se renderiza desde input de usuario (XSS).
  • Backups de Supabase verificados, no asumidos.
conectores / MCP recomendados

Los MCP le dan a Claude acceso a herramientas externas. Se agregan con /mcp en Claude Code o desde Ajustes → Conectores en Claude.ai.
